# MFCB Platform — Endpoint Mappings & Server Actions

> Version 1.0 | 2026-05-06
> Base URL (production): `https://api.mfcb.co.bw/v1`
> Base URL (sandbox): `https://sandbox-api.mfcb.co.bw/v1`
> Portal base: `https://portal.mfcb.co.bw`

---

## Conventions

| Symbol | Meaning |
|--------|---------|
| `[M]` | Mandatory field in request |
| `[C]` | Conditional field |
| `[O]` | Optional field |
| `→ Screen` | Portal screen this action serves |
| `← Event` | Audit/security event written on execution |

All mutating requests require:
- `Authorization: Bearer {token}` header
- `Idempotency-Key: {uuid}` header (POST/PATCH/PUT)
- `X-Correlation-Id: {uuid}` header (auto-generated by gateway if absent)

## Section 1 — Authentication

### 1.1 `POST /oauth/token`

**Purpose:** Issue access token for API clients using client credentials flow.

**Auth:** None (public endpoint — client credentials in body)

**→ Screen:** API client integration (no portal screen)

**Request:**
```
Content-Type: application/x-www-form-urlencoded

grant_type=client_credentials   [M]
client_id={uuid}                [M]
client_secret={string}          [M]
scope={space-separated scopes}  [O] defaults to all granted scopes
```

**Response `200`:**
```json
{
  "access_token": "eyJ...",
  "token_type": "Bearer",
  "expires_in": 3600,
  "scope": "submit:data read:batch read:errors"
}
```

**Response `401`:**
```json
{
  "error": "invalid_client",
  "error_description": "Client authentication failed."
}
```

**Server Actions:**
1. Validate `client_id` exists and `status = ACTIVE` in `api_clients`
2. Verify `client_secret` against stored Argon2id hash
3. Check `expires_at` if set — reject if expired
4. Check request IP against `api_clients.allowed_ip_ranges`
5. Intersect requested scopes with `api_clients.scopes`
6. Sign JWT with institution_id, client_id, granted scopes, expiry
7. Write `audit_logs` action=`TOKEN_ISSUED`

**← Events:** `TOKEN_ISSUED` (SUCCESS), `AUTH_FAILURE` (FAILURE)

---

### 1.2 `POST /portal/auth/login` *(Portal Server Action)*

**Purpose:** Portal user password + MFA verification.

**→ Screen:** 1.1 Login, 1.2 MFA

**Step 1 — Password:**
```json
{ "email": "user@bank.co.bw", "password": "..." }
```

**Step 2 — MFA (issued after step 1 success):**
```json
{ "session_token": "{temp_token}", "otp_code": "123456" }
```

**Server Actions (Step 1):**
1. Look up `users` by email; return generic error if not found (prevent enumeration)
2. Verify password hash
3. Check `status` = `ACTIVE`; check `locked_until` not in future
4. Increment `failed_login_count` on failure; lock if ≥ 5
5. On success: issue short-lived `session_token` (5 min) scoped to MFA step
6. Write `audit_logs` action=`LOGIN_ATTEMPT`

**Server Actions (Step 2):**
1. Verify `session_token` not expired
2. Validate TOTP code against `mfa_secret_encrypted` (decrypt → verify)
3. Check `mfa_enrolled = TRUE`; if FALSE redirect to enrollment
4. On success: issue full session cookie (HttpOnly, Secure, SameSite=Strict)
5. Write `audit_logs` action=`LOGIN_SUCCESS`

**← Events:** `AUTH_FAILURE`, `MFA_FAILURE`, `ACCOUNT_LOCKED`, `LOGIN_SUCCESS`

---

### 1.3 `POST /portal/auth/mfa-enroll` *(Portal Server Action)*

**Purpose:** Complete TOTP enrollment.

**→ Screen:** 1.4 MFA Enrollment

**Request:**
```json
{ "totp_secret": "{server_generated}", "verification_code": "123456" }
```

**Server Actions:**
1. Validate TOTP code against secret
2. Encrypt secret with AES-256, store in `users.mfa_secret_encrypted`
3. Set `users.mfa_enrolled = TRUE`
4. Write `audit_logs` action=`MFA_ENROLLED`

---

### 1.4 `POST /portal/auth/password-reset/request` *(Portal Server Action)*

**→ Screen:** 1.3 Forgot Password

**Server Actions:**
1. Look up user by email (always return 200 — no enumeration)
2. If found: generate single-use token (SHA-256 random, expires 60 min), store hashed
3. Send reset email with token URL
4. Write `audit_logs` action=`PASSWORD_RESET_REQUESTED`

---

### 1.5 `POST /portal/auth/password-reset/confirm` *(Portal Server Action)*

**→ Screen:** 1.3 Reset Password (step 2)

**Request:**
```json
{ "token": "{reset_token}", "new_password": "...", "confirm_password": "..." }
```

**Server Actions:**
1. Validate token: exists, not expired, not already used
2. Validate password strength (min 10 chars, 1 upper, 1 number, 1 special)
3. Confirm passwords match
4. Hash and store new password; invalidate token; invalidate all active sessions
5. Write `audit_logs` action=`PASSWORD_RESET_COMPLETED`

### 2.5 `POST /institutions/{institution_id}/api-clients` *(Portal Server Action)*

**Auth:** `SUPER_ADMIN`, `BUREAU_OPS`

**→ Screen:** 9.4 API Client Management

**Request:**
```json
{
  "scopes": ["submit:data", "read:batch", "read:errors"], [M]
  "allowed_ip_ranges": ["196.x.x.x/32"],                  [O]
  "token_ttl_seconds": 3600,                               [M]
  "expires_at": "2027-01-01T00:00:00Z"                    [O]
}
```

**Response `201`:**
```json
{
  "client_id": "uuid",
  "client_secret": "raw_secret_shown_once",
  "scopes": [...],
  "expires_at": "..."
}
```

**Server Actions:**
1. Generate UUID `client_id`
2. Generate cryptographically random 64-char `client_secret`
3. Hash secret with Argon2id; store hash only
4. Return plain secret in response body once; never retrievable again
5. Write `audit_logs` action=`API_CLIENT_CREATED`

### 3.2 `POST /batches/file`

**Purpose:** Upload CSV, Excel, or fixed-length Form G file.

**Auth:** `submit:data` scope

**→ Screen:** 3.2 New Batch Upload (file path)

**Request:** `multipart/form-data`

```
supplier_reference_number: BW0001        [M]
reporting_month: 20260331                [M]
file_type: LIVE                          [M]
sequence_number: 1                       [M]
file: {binary}                           [M] — .txt / .csv / .xlsx max 50 MB
```

**Response `202`:**
```json
{
  "batch_id": "bat_uuid",
  "status": "QUEUED",
  "file_name": "BW0001_ALL_L702_M_20260331_1_1.txt",
  "file_hash_sha256": "abc123...",
  "file_size_bytes": 2048000,
  "correlation_id": "..."
}
```

**Server Actions:**
1. Authenticate; enforce institution match
2. Virus scan file (async; reject if threat found)
3. Compute SHA-256 hash; check for duplicate file hash (warn, not reject)
4. Validate file name against naming convention regex
5. Parse file format (detect fixed-length / CSV / Excel)
6. Validate header record: `H` type, SRN, month-end, version, creation date
7. Validate trailer: `T` type, record count matches data record count
8. Persist `batch_uploads`, `raw_submission_records`
9. Enqueue validation job
10. Write `audit_logs` action=`BATCH_FILE_UPLOADED`

**← Events:** `BATCH_FILE_UPLOADED`, `INVALID_FILE_NAMING` (if naming check fails)

## Section 4 — Validation Engine (Internal — No Direct API)

The validation engine runs asynchronously after batch ingestion. It executes these check stages in order:

| Stage | Check Type | Description |
|-------|-----------|-------------|
| 1 | File-level | Header/trailer present; SRN match; record count match; file naming |
| 2 | Mandatory fields | All `M` fields populated per Form G layout |
| 3 | Format validation | Alpha/numeric format, length, CCYYMMDD dates |
| 4 | Reference table | Each code field value exists in `reference_codes` and is not deprecated |
| 5 | Date logic | Not in future; not older than 36/60 months; sequence relative to account opened date |
| 6 | Financial logic | Opening balance / current balance / instalment / overdue rules by account type and status |
| 7 | Status code rules | Status allowed for account type (via `status_account_type_rules`); status date rules |
| 8 | Cross-field | Deferred date ↔ payment type; joint loan count ↔ ownership type; third party ↔ status A |
| 9 | Duplicate check | SRN + account + sub-account + branch unique within batch; idempotency key; LIVE sequence |
| 10 | Historical consistency | Last payment date not earlier than prior submission; status progression logic |

**On completion:** Update `batch_uploads.status`, `accepted_count`, `rejected_count`. Write all errors to `validation_errors`. Trigger webhook if configured.

### 5.1 `POST /credit-reports/search`

**Purpose:** Search for a borrower and initiate a credit report.

**Auth:** `read:reports` scope; institution isolation

**→ Screen:** 5.1 Borrower Search

```json
  "date_of_birth": "19850315",          [C] — required for SURNAME_DOB
  "inquiry_reason": "CREDIT_APPLICATION", [M]

```

**Response `200` (match found):**
```json
{
  "result": "MATCH",
  "report_id": "rpt_uuid",
  "borrower_id": "borr_uuid",
  "inquiry_id": "inq_uuid",
  "correlation_id": "..."
}
```

**Response `200` (no match):**
```json
{
  "result": "NO_MATCH",
  "inquiry_id": "inq_uuid",
  "message": "No record found for the supplied search criteria.",
  "correlation_id": "..."
}
```

**Response `200` (multiple candidates):**
```json
{
  "result": "MATCH_REVIEW_REQUIRED",
  "inquiry_id": "inq_uuid",
  "message": "Multiple potential matches found. Manual review required.",
  "correlation_id": "..."
}
```

**Server Actions:**
1. Validate search params; validate at least one search field present
2. Validate `inquiry_reason` non-empty
3. Validate `customer_consent_reference` present (unless exempt reason)
4. Write `credit_inquiries` record BEFORE search executes (immutable audit)
5. Write `audit_logs` action=`REPORT_REQUESTED`
6. Execute matching logic against `borrowers` and `borrower_identifiers`
7. If MATCH: generate `report_id`, assemble report asynchronously, return immediately
8. If NO_MATCH: log, return result — no error
9. If MATCH_REVIEW_REQUIRED: log, flag for bureau ops manual review
10. Return result with `inquiry_id` for reference

**← Events:** `REPORT_REQUESTED`

1. Resolve `report_id` → `inquiry_id` → validate institution match
2. Assemble report from `borrowers`, `borrower_identifiers`, `borrower_addresses`, `borrower_employment`, `credit_accounts`, `repayment_history`, `account_status_events`, `credit_inquiries`
3. Filter `credit_accounts` to only account types in `token.institution.enabled_products` (data minimisation — SEC-009)
4. Mask Omang/Passport (last 4 visible), mask account numbers (last 4), null-out `income`
5. Limit repayment history to 24 months (or institutional permission)
6. Limit adverse events to 2 years
7. Write `audit_logs` action=`REPORT_RETRIEVED`

### 7.2 Webhook Delivery (Internal — Background Job)

Triggered automatically after batch processing completes.

**Delivery Rules:**
- Sign payload: `X-MFCB-Signature: HMAC-SHA256={hex}`
- Expect HTTP 200 from institution endpoint within 10 s
- On failure: retry 3× with exponential backoff (30 s, 90 s, 270 s)
- After 10 consecutive failures: set `webhook_configs.status = FAILED`, alert Bureau Ops
- All deliveries logged to `audit_logs`

## Section 10 — HTTP Status Code Reference

| Code | When Used |
|------|-----------|
| `200` | Successful GET or synchronous action |
| `201` | Resource created (institution, api_client) |
| `202` | Batch accepted for async processing |
| `400` | Malformed JSON, missing required top-level fields, unsupported file type |
| `401` | Missing, expired, or invalid access token |
| `403` | Token valid but scope/institution permission denied |
| `404` | Resource not found |
| `409` | Idempotency key conflict; duplicate LIVE submission; conflicting state |
| `422` | Structurally valid request but content fails validation rules |
| `429` | Rate limit exceeded |
| `500` | Unhandled server error |
| `503` | Scheduled maintenance or temporary outage |

**Standard Error Response Format:**
```json
{
  "status": "VALIDATION_FAILED",

      "severity": "REJECT",
      "message": "Omang must be 9 numeric characters and must not be a repeated dummy value.",
      "raw_value": "111111111"
    }
  ]
}
```

## Section 12 — Idempotency Key Rules

| Scenario | Behaviour |
|----------|-----------|
| Same `Idempotency-Key` within 24 h | Return original response; do not reprocess |
| Same `Idempotency-Key` after 24 h | Treat as new request |
| LIVE batch: same SRN + reporting_month + sequence_number | Return `409 DUPLICATE_LIVE_SUBMISSION` |
| LIVE batch: same SRN + reporting_month but higher sequence_number | Accept as resubmission |
| TEST batch: any re-submission | Always accept; TEST batches do not have duplicate protection |
